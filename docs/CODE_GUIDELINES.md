## The `atrium` build pipeline (mental model)

`atrium image --config build.json` ([`image.rs`](../tools/atrium-tool/src/image.rs) `run`) runs one pipeline:

```
base copy â†’ grow â†’ harvest apps â†’ MG/LaunchBox enrich â†’ compatibility merge
          â†’ filter_present_apps â†’ bake art â†’ catalog (paged + legacy) â†’ install launcher
```

Two behaviors to internalize:

- **The build never mutates `data/*.jsonl`.** It works off a throwaway copy in a
  stage dir. Edit the real dataset separately; a build only reads it.
- **`filter_present_apps`** ([image.rs](../tools/atrium-tool/src/image.rs)) drops
  any catalog record whose `app` file isn't actually on the volume. So
  over-listing is safe â€” but a title won't appear unless its bits truly landed.
  This keeps the on-screen catalog in lockstep with the disk.

## The content model (how a title flows)

Data is layered; each stage has one job (full detail in
[data/README.md](../data/README.md) and [docs/06](06-content-pipeline.md)):

| File | Role | Editable by hand? |
|------|------|-------------------|
| `data/library.jsonl` | **Generated** identity + descriptive metadata (id/name/kind/year/vendor/genre/`app`/`harvest_src`) | No â€” regenerated by `library scan` |
| `data/curated.jsonl` | Hand overlay: non-MacPack add-ons (full records) + corrections; **curation wins** | Yes |
| `data/compatibility.jsonl` | Per-title requirement facets (`color`/`mouse`/`maxDepth`/`minOS`/`maxOS`/`minMem`/`minCPU`/`arch`); hand-verified wins | Yes |
| `data/categories.jsonl` | Editable category DB (many-to-many membership); `library categorize` seeds it, hand/GUI edits preserved | Yes |
| `data/taxonomy.json` | Canonical category order + seed maps + `recommended` list | Yes |
| `data/targets.json` | Named machine profiles (base OS + art depths + RAM + disk size) | Yes |
| `data/donors.json` | Donor **key â†’ image path** (machine-local, gitignored); records reference the *key* so the dataset stays portable | Yes (local) |

- **Selection**: a build picks a subset via `Selection` ([config.rs](../tools/atrium-tool/src/config.rs)): `List{ids}` / `Categories{â€¦}` / `All`. Each selected record is sourced from its `harvest_src: {donor, path}` (donor key resolved through `donors.json`).
- **The on-Mac catalog is derived, never hand-authored** â€” a paged tree (`metadata/index.jsonl` + `cats/<slug>.jsonl` + `hotkeys.jsonl`) generated from the dataset + facets + taxonomy ([docs/21](21-category-paging.md)).

## Invariants & gotchas (the expensive-to-learn list)

- **HFS bootable volume â‰¤ 2 GB.** Capped by `MAX_DISK_MB` ([config.rs](../tools/atrium-tool/src/config.rs)); build larger and the disk **hangs on a dark screen at boot with no error**.
- **HFS names â‰¤ 31 chars.** Folder/file names truncate on the volume (e.g. `â€¦America's Past Is Carmen Sandiego` â†’ `â€¦America's Past Is Carm`) â€” which once made two Carmen folders collide. Guards: `fs_id()` keeps baked art filenames â‰¤ 31; **`config::hfs_name()`** shortens a title to a *distinct* â‰¤ 31-char on-volume folder/app name (use it whenever the tooling **creates** a name from a title, so nothing is silently mangled); and the catalog generator **warns** on any `app` path component > 31 chars. Catalog `app` paths must match the on-disk (possibly shortened) names.
- **On-device catalog format is strict** ([catalog.rs](../tools/atrium-tool/src/catalog.rs)): **MacRoman** encoding, **CR** line endings, `TEXT` type; â‰¤ **256** items in a single-file catalog (`MAX_ITEMS`), â‰¤ **128**/page (`MAX_CAT_ITEMS`), â‰¤ **8** categories/item, fixed field lengths. The generator warns/clamps rather than silently overflowing the 68k reader. **`MAX_ITEMS` is not a library cap** â€” `run_paged` deliberately exceeds it across pages, and only ONE page is ever resident, so a paged build has no total-title limit and its launcher RAM doesn't scale with the collection (the real terms are page size and *art depth*; see [docs/21](21-category-paging.md)).
- **`harvest` renames folders and re-picks the launch app.** `harvest_one` ([harvest.rs](../tools/atrium-tool/src/harvest.rs)) chooses an `APPL` via `pick_appl` and **renames the destination folder to that APPL**. That's right for messy MacPack donor disks, but for *already-installed* content (e.g. a hand-built donor reservoir) it will override curated `app` paths and can mis-pick an installer. To place already-installed folders verbatim, copy them with `rb-cli cp` â€” **do not** route them through `harvest`.
- **The launcher is NOT embedded in `atrium`.** A build reads `build/MacAtrium.bin` (or `$MACATRIUM_LAUNCHER`, or the config `launcher`) â€” see `launcher_bytes()` in [config.rs](../tools/atrium-tool/src/config.rs). (The tool README says "embedded"; that's stale.) The *data* files **are** embedded (`EMBEDDED_LIBRARY`/`COMPAT`/`TAXONOMY`/`CATEGORIES`/`TARGETS`), so a build needs no data paths unless overriding.
- **Stale `rb-cli` on `$PATH` trap.** A bare `"rb-cli"` can silently shadow the configured binary and write a corrupt catalog. `resolve_bin` prefers an **absolute** path and `log_version` prints which binary/version actually ran â€” always configure an absolute `rb_cli` and glance at that line.
- **Classic Mac apps have 0-byte data forks** (code lives in the resource fork). `rb-cli ls` shows the *data* fork only, so summing `ls` sizes wildly undercounts â€” use `rb-cli show fs-info` for true (both-fork) volume usage, or `get-binhex` to gauge one app's resource fork.
- **Don't `git stash` for before/after image builds.** Stashing source, rebuilding, then popping without rebuilding again produces a mismatched binary and hours of phantom debugging. Use a separate output path or a worktree.

## Conventions

- **`BuildConfig` is the one build model** ([config.rs](../tools/atrium-tool/src/config.rs)): the CLI deserializes it from `--config` JSON, the GUI builds it field-by-field. Add build knobs as `BuildConfig` fields, not ad-hoc CLI args.
- **The bundled âŠ• user layering pattern** recurs â€” `Target`s are bundled (`data/targets.json`, embedded) overlaid with user entries in `~/.macatrium.json` ([targets.rs](../tools/atrium-tool/src/targets.rs), [settings.rs](../tools/atrium-tool/src/settings.rs)). New named, shareable things should mirror it.
- **JSONL everywhere tolerates comments**: `#` / `//` and blank lines are skipped; one record per line, keyed by `id`.
- **IDs are slugs** via `harvest::slugify` (lowercase ASCII, accents folded, non-alnum â†’ `-`).
- **Overlays merge by id, overlay wins** ([merge.rs](../tools/atrium-tool/src/merge.rs)) â€” the mechanism for corrections and for appending new records (`--fill-missing` flips to gaps-only).
- **Adding an `atrium` verb**: a `Cmd` variant in [main.rs](../tools/atrium-tool/src/main.rs) (clap) dispatching to a small module fn. Keep modules focused and unit-tested â€” `catalog`/`harvest`/`selection` have good test patterns to copy.
- **Fail soft on data**: one bad JSONL line shouldn't abort a run (skip + warn); use `anyhow` with `.with_context(...)`.

## Building, running, verifying

- **`atrium` (host tool)** â€” pure Rust, builds anywhere:
  `cd tools/atrium-tool && cargo build --release && cargo test`
- **The launcher (68k)** â€” Retro68/CMake: `cmake --build build -j` â†’ `build/MacAtrium.bin`; portable C-core tests: `cd tests && make && ./host_test`. See [docs/04](04-toolchain-build.md).
- **`rb-cli`** (rusty-backup) is the HFS-image I/O layer the tool shells out to; it's a separate binary.
- **Headless verify** in the Snow harness (Mac II, 8-bit) or QEMU `q800` (68040/colour). See [docs/44](44-memory-and-art-modes.md) for `MEM_DEBUG` and the harness READMEs.
- On this repo's primary machine these run under **WSL** while driven from Windows â€” the machine-specific operating notes (paths, WSL invocation quirks, git) live in the root [`CLAUDE.md`](../CLAUDE.md).
