## 2. THE BLOCKER — full 24-bit Quadra build fails on an rb-cli catalog bug

`atrium image` of the full library (`selection:all`, 7.5.5, art `1,8,24`,
`max_art_size 448`, `app_mem_kb [3584,3072]`, `disk_size_mb 2000`,
`rb_cli=…/target/release/rb-cli`) **deterministically dies at ~17,934 catalog records**
with `create_file: disk full: no free B-tree nodes`, and `fsck` shows **415
`IndexSiblingLinkBroken`** (1.14 nodes/rec, 776 MB data still free).

Established this session:
- The **starting template is fsck-CLEAN** (`MacLC_7-5-5.hda@1`, 265 files, 0 errors) →
  the corruption is **GENERATED by current rb-cli (1b22518) during injection**, NOT
  inherited. cfg.out is the only corrupt disk.
- The build injects app files via **`put-binhex --clear-inited`** (both forks + Finder
  flags), **not** `put-macbinary --force`, onto an **`expand`-clone** of the clean base
  (cp template → out → `expand` 73 MB→2000 MB → inject). No `rm`/`delete`, no `--force`
  overwrites, no dup dest paths; harvest **skips** intra-app duplicate-leaf-name files.
- The corrupting combo = *expand-clone base + put-binhex --clear-inited + real
  resource forks (some 12 MB) + ~18k records + skip-on-collision*. Each factor ALONE
  is clean (ruled out: plain put flat/multi-parent/shuffled, put+setrsrc,
  put-into-expanded, put-binhex ≤2.5k, put-macbinary).

## 3. 2 GB boot cap → multi-volume is a backlogged feature (docs/23)

**Sad Macs** (`0000000F/00000004`). So boot volumes are capped at 2 GB. The full-res
